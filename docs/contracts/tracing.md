# Contract: Tracing (ADR-0032 — Postgres-first, behind a `TraceSink` port)

Version: 1.0.0. Two independent, duplicated implementations (ADR-0008 — never shared): each of
the two owned LangGraph graphs traces itself through its own service's `TraceSink` port.

| Graph | Service | Table | Schema owner |
|---|---|---|---|
| `get_task_context` (+ `kb_search`) | `services/mcp-server` | `trace_span` | kb-builder Alembic migration `0021_trace_span` (Knowledge Registry, public schema) |
| review-panel draft engine | `services/review-panel` | `trace_span` | the service's own idempotent bootstrap DDL, dedicated `review_panel` schema (documented Alembic exemption, `review-panel.md`) |

No table, DTO, or Python type is shared between the two services. This document is the only
thing that keeps their two `trace_span` tables and two `TraceSink` ports aligned in shape.

## Why (one line)

Both graphs are trace-*ready* but nothing was ever activated; LangSmith (committed on paper by
ADR-0030 §4) is withdrawn before ever shipping — a paid, non-Azure hosted tracing SaaS the owner
explicitly rejected. Per-step visibility (which node was slow, whether the retry fired, how long
each reviewer lens took) is exactly the ledger pattern this platform already owns: Postgres rows,
no vendor. See `docs/adr/0032-postgres-first-tracing-behind-a-port.md` for the full decision.

## Span shape

One row = one completed unit of work (a whole tool call, or one node inside its graph).

| Column | Type | Notes |
|---|---|---|
| `span_id` | `uuid`, PK | Generated by the **application** (`uuid4()`) at span-creation time — the column carries no server-side default; the DTO always supplies it. |
| `trace_id` | `text`, not null | Groups every span belonging to one logical unit. mcp-server mints a fresh `uuid4().hex` per tool call (one call = one trace). review-panel uses the run's `draft_key` (`<repo>#<pr>@<head_sha>`) — deterministic and stable across a crash + resume, so a resumed attempt's spans correlate with the interrupted attempt's under the same trace. |
| `parent_span_id` | `uuid`, nullable | `NULL` on a root span. Every other span in the trace carries the root's `span_id`. review-panel may show **more than one** root (`parent_span_id IS NULL`) per `trace_id` when a run is resumed after a crash — one root per process **attempt**, not one per draft. |
| `name` | `text`, not null | Stable operation name. mcp-server: `get_task_context` (root), `resolve_scope` / `blast_radius` / `conventions` / `similar_prior_changes` / `synthesize` / `broaden` (nodes), `kb_search` (its own root, no children). review-panel: `review_panel.draft_run` (root), `load_pr`, `review_<lens>` (`review_bug`/`review_security`/`review_quality`/`review_test_coverage`), `reconcile`, `store_draft`. |
| `service` | `text`, not null | Literal `"mcp-server"` or `"review-panel"` — never derived from a request field. |
| `started_at` / `ended_at` | `timestamptz`, not null | Wall-clock window of the traced operation (converted from `time.monotonic()` measurements taken at the same call using one fixed wall-clock/monotonic reference pair — monotonic values are never stored directly, they have no fixed epoch across processes). |
| `status` | `text`, not null, `CHECK IN ('ok', 'error')` | `error` iff the traced operation raised. |
| `attributes` | `jsonb`, nullable | Aggregate-only metadata: counts, booleans, enum-valued labels (`lens`, `verdict`, `budget_status`), token counts, result sizes. **Never** free text, prompts, document bodies, or query text — see "No-content rule" below. |
| `created_at` | `timestamptz`, not null, `server_default now()` | Row-insert time (observability of the writer itself, not part of the span's own window). |

## Port semantics — emit-after-completion

Spans are observability, not control flow, so the port is a single emit call over an
**already-complete** span — there is no `start_span()`/`end_span()` pairing to keep alive across
an `await`, no thread-local, and nothing a node must remember to close:

```python
class TraceSink(Protocol):
    async def emit(self, span: Span) -> None: ...
```

The caller measures `started_at`/`ended_at` (and `status`) itself around the unit of work it is
already timing (mcp-server's node functions already compute this window for the ledger's
`node_latency_ms`; review-panel's node closures wrap their own body), builds one `Span`, and hands
it to the sink whole, after the work is done. This is deliberately the simplest correct shape —
nothing a partial/interrupted span could ever need to reconcile.

### The fail-soft boundary: `emit_span`

Every caller goes through one function, not `sink.emit()` directly, so fail-soft is enforced in
exactly one place per service rather than re-implemented by every adapter:

```python
async def emit_span(sink: TraceSink, span: Span) -> None:
    try:
        await sink.emit(span)
    except Exception:
        logger.warning("event=trace_sink_error name=%s service=%s status=%s", ...)
        # never raises — the span is dropped, the traced call is unaffected
```

- A dead/slow/misbehaving sink **never** raises into the traced call, never delays it materially
  (the emit happens after the call's own work is already done — see "Placement" below), and never
  charges budget (tracing touches no budget meter, ever).
- A concrete `TraceSink` implementation (e.g. `PostgresTraceSink`) is free to raise on a genuine
  failure — the contract is enforced at the `emit_span` boundary, not by requiring every adapter to
  self-swallow. This is also how the "raising sink doesn't fail the call" test works: a
  deliberately-raising fake sink proves the boundary holds regardless of which adapter is behind it.

### Placement (when `emit_span` is actually called)

- **mcp-server / `get_task_context`**: the graph already collects one `NodeSpan(node, started,
  ended)` per node into run telemetry (used today for the ledger's `node_latency_ms`). The tool
  entry point (`task_context.get_task_context`), after the graph run completes, converts that
  telemetry plus one synthesized root span into `Span` rows and emits each — the same place the
  `retrieval_event` ledger row is already written. Graph node code
  (`context_broker/task_context_nodes.py`) is untouched: it depends on nothing tracing-related.
- **mcp-server / `kb_search`**: one span per call, emitted at the tool's single exit point
  (success, budget-denied, or exception) — no children, no LangGraph involved.
- **review-panel**: the root span wraps the whole `graph.ainvoke(...)` call in
  `application.compute_draft`. Each node (`load_pr`, `review_<lens>`, `reconcile`, `store_draft`)
  emits its own span from inside its own closure, using a root `span_id` and a `TraceSink` handed
  to it via `PanelDependencies` (constructed once per draft-run attempt) — **never** through
  `PanelState`, so a span is never part of what the LangGraph checkpointer persists (see
  "Crash-resume" below).

## Fail-soft rule (ADR-0032 §3)

Tracing never fails, materially delays, or budget-charges the call it observes. Concretely:

1. A sink exception is caught at the `emit_span` boundary, logged as a structured
   `event=trace_sink_error` line (name/service/status only — never span attributes, which could
   carry request-shaped data), and the span is dropped.
2. `emit_span` is always called **after** the traced work's own result is already computed (or the
   exception is already known) — a slow sink cannot slow down the answer the caller is about to
   return; the emit is the last thing that happens on that path, and its own failure doesn't
   unwind anything already decided.
3. No budget meter (`kb_search`'s per-window cap, `get_task_context`'s response-token cap) ever
   reads or writes anything through tracing. Emitting a span costs the call nothing tokens-wise.

## Env selection (`TRACE_SINK`)

| Value | Sink |
|---|---|
| unset / `postgres` | `PostgresTraceSink` when a database is configured, else `NullTraceSink` |
| `none` | `NullTraceSink` unconditionally |
| anything else | a configuration error (fails at startup, not silently) |

mcp-server: selected in `context_broker/dependencies.py` (`select_trace_sink`), mirroring how
`SearchClient`/`EntailmentClient` selection is centralized — graph code and tool modules receive a
constructed `TraceSink` through `BrokerDeps.trace_sink` and never read the env themselves.
`BrokerDeps.trace_sink` itself defaults to `NullTraceSink()` (the safe default for any caller that
builds `BrokerDeps` directly, e.g. tests); the "postgres when a database is configured" default is
applied once, at the real server's startup wiring (`mcp/server.py: create_app`).

review-panel: selected in `cli.py`'s `_compute` (the same place `REVIEW_PANEL_DATABASE_URL` is
already read to decide between the Postgres checkpointer/draft-store and the in-memory fallback) —
`TRACE_SINK=postgres` with no `REVIEW_PANEL_DATABASE_URL` configured falls back to
`NullTraceSink`, exactly like the checkpointer/draft-store fallback already does, and is logged
the same way (`event=persistence_fallback`-style plain log, never silent).

## Table ownership

- **mcp-server**: `trace_span` lives in the Knowledge Registry (public schema), migrated by
  kb-builder (`0021_trace_span`, reversible). mcp-server only ever `INSERT`s into it (no reads —
  tracing is a write-only sink from the tool's point of view; querying spans is an operator
  concern, same posture as `retrieval_event` today, and a future dashboard tile per ADR-0014's
  consequences section reads it directly).
- **review-panel**: `trace_span` lives in the dedicated `review_panel` schema, created idempotently
  (`CREATE TABLE IF NOT EXISTS`) alongside `review_draft`, under the same documented Alembic
  exemption (`review-panel.md` — derived, recomputable, prunable state; rollback is
  `DROP SCHEMA review_panel CASCADE`).

## No-content rule

Same aggregate-only posture as the ADR-0014 dashboard: **never** `query_text`, `task_description`,
`normalized_query`, `body_text`/diff/PR text, prompts, or any other raw/untrusted content field, in
`attributes` or anywhere else on a span. `Span.__post_init__` rejects construction if `attributes`
carries any of a fixed set of forbidden keys (`query_text`, `normalized_query`, `task_description`,
`query`, `prompt`, `body_text`, `content`) — a coding-time contract violation, not a request-time
one (nobody controls `attributes` from outside the process). A hermetic test in each service
constructs a `Span` with a forbidden key and asserts the rejection.

## Testing conventions

- **mcp-server**: spans written per node against a real, migrated `TEST_DATABASE_URL`
  (`tests/integration/test_tracing.py`); a deliberately-raising fake sink proves the tool call (and
  its `retrieval_event` "approved" ledger row) is unaffected
  (`tests/integration/test_tracing_fail_soft.py` or folded into the same module);
  `NullTraceSink` is asserted as `BrokerDeps`'s bare default; an `InMemoryTraceSink` fake proves the
  port swap (the designated Langfuse-later seam) without touching Postgres at all; the static
  no-content test needs no database.
- **review-panel**: the same shape, with `InMemoryTraceSink`/a raising fake standing in for
  Postgres in the hermetic suite (`tests/panel_test_support.py`), and a `TEST_DATABASE_URL`-gated
  test proving `PostgresTraceSink` actually lands rows in the `review_panel` schema.
